    Wednesday: To Do List Finding and Saving Objects

    Over the next few lessons, we will add more features to our To Do List application. We'll focus particularly on the static variables and methods we just learned about. More specifically, we'll add the following to our To Do List:

    • A mCompleted boolean property to record whether a Task is completed.
    • A mCreatedAt property to record when a Task was created.
    • A static variable and method to record and return all Task objects.
    • A static method to empty the list of Task objects.
    • A mId property to contain a unique ID for each Task.
    • A method that locates a specific Task using this unique ID.

    Recording Whether a Task Has Been Completed

    First, let's add a property that tracks whether a Task is complete. It should default to false when a Task is created. Then, when a user completes the Task, it will be updated to true.

    As always, we'll begin with a JUnit test. First, we'll test that mCompleted is set to false by default when a Task is instantiated:

    to-do/src/test/java/TaskTest.java
    ...
@Test
public void isCompleted_isFalseAfterInstantiation_false() {
  Task myTask = new Task("Mow the lawn");
  assertEquals(false, myTask.isCompleted());
}
...

    After confirming the test fails, we'll declare the mCompleted property, set it to false in the Taskconstructor, and create a corresponding isCompleted() getter method:

    to-do/src/main/java/Task.java
    public class Task {
  private String mDescription;
  private boolean mCompleted;

  public Task(String description) {
    mDescription = description;
    mCompleted = false;
  }

  public String getDescription() {
    return mDescription;
  }

  public boolean isCompleted() {
    return mCompleted;
  }
}

    As you can see, defining a member variable in the constructor without taking it as an argument allows us to define a default value. That is, all new Task objects will have an mCompleted property set to false, because our constructor won't allow us to set it to anything else. And, after implementing this code, our tests should pass.

    Dates and Timestamps

    Next, we want to record when a new Task is created. We'll need a property to automatically record the date and time our program constructs a Task. As always, we'll begin with a test:

    to-do/src/test/java/TaskTest.java
    import java.time.LocalDateTime;
...
@Test
public void getCreatedAt_instantiatesWithCurrentTime_today() {
  Task myTask = new Task("Mow the lawn");
  assertEquals(LocalDateTime.now().getDayOfWeek(), myTask.getCreatedAt().getDayOfWeek());
}
...

    We haven't worked with date types in Java yet, but thankfully it's fairly straightforward!

    • As you can see this test uses the LocalDateTime class, so we must import it at the top of the file.
    • LocalDateTime.now() returns a timestamp of the current time.
    • Chaining .getDayOfWeek() after LocalDateTime.now() returns only the day of the week denoted by the timestamp.

    We'll add code to Task.java file to automatically record the time a Task is created:

    to-do/src/main/java/Task.java
    import java.time.LocalDateTime;

public class Task {
  private String mDescription;
  private boolean mCompleted;
  private LocalDateTime mCreatedAt;

  public Task(String description) {
    mDescription = description;
    mCompleted = false;
    mCreatedAt = LocalDateTime.now();
  }

  public String getDescription() {
    return mDescription;
  }

  public boolean isCompleted() {
    return mCompleted;
  }

  public LocalDateTime getCreatedAt() {
    return mCreatedAt;
  }
}

    We've added a private member property (mCreatedAt) to our class, set its value in the constructor using LocalDateTime.now(), and created a getter method to access it. The test should now pass.

    We'll work more with date objects and timestamps in the future, but if you'd like to explore these concepts now check out the Java Documentation on Date Classes (Links to an external site.)Links to an external site.

    Retrieving a List of All Objects

    Next, we want functionality to retrieve all instances of a class. Just like we saw in the previous lesson, we'll define a static ArrayList to keep track of each object our class creates. Then, we'll create a static method to return it. Let's begin with a test:

    to-do/src/test/java/TaskTest.java
    ...
  @Test
  public void all_returnsAllInstancesOfTask_true() {
    Task firstTask = new Task("Mow the lawn");
    Task secondTask = new Task("Buy groceries");
    assertEquals(true, Task.all().contains(firstTask));
    assertEquals(true, Task.all().contains(secondTask));
  }
...

    We use two assertions to ensure the ArrayList returned by Task.all() includes both tasks. Since the purpose of all() is to return all Tasks, we must test it in a manner that confirms the method not only returns multiple Task objects, but every Task object created thus far.

    After confirming the test fails we can implement this functionality:

    to-do/src/main/java/Task.java
    import java.time.LocalDateTime;
import java.util.ArrayList;
import java.util.List;

public class Task {
  private String mDescription;
  private boolean mCompleted;
  private LocalDateTime mCreatedAt;
  private static List<Task> instances = new ArrayList<Task>();

  public Task(String description) {
    mDescription = description;
    mCompleted = false;
    mCreatedAt = LocalDateTime.now();
    instances.add(this);
  }

  public String getDescription() {
    return mDescription;
  }

  public boolean isCompleted() {
    return mCompleted;
  }

  public LocalDateTime getCreatedAt() {
    return mCreatedAt;
  }

  public static List<Task> all() {
    return instances;
  }
}

    In the code above, we've do the following:

    • Import the necessary List and ArrayList classes.

    • Create a private static variable ArrayList named instances. This will eventually contain every Task object we create.

    • Add a public static method to retrieve the list of all Tasks. Because this method is static, it must be called upon the Task class itself; not a particular instance. Like this: Task.all()

    • Add every new Task object to our static instances ArrayList with the following line: instances.add(this);. This logic resides in the constructor because the constructor is called every time we create a Task. This ensures every new Task will be included in instances.

    Now, our Task.all() test should pass.

    Clearing the List of All Tasks

    Next, we need the ability to clear our list of all Tasks. That is, empty the instances ArrayList.

    This functionality is incredibly important during development. We'll need to empty our list between each JUnit test. Otherwise, our list will continually grow as multiple tests create multiple Tasks. If one test creates a few Tasks, the test following test will have to maneuver around leftover objects. This can cause issues, errors, and test failures. In order to minimize room for error, we want to ensure we have a "fresh slate" before each test.

    We'll define a static clear() method to empty the instances ArrayList. But, as always, we'll write a test first:

    to-do/src/test/java/TaskTest.java
    ...
@Test
public void clear_emptiesAllTasksFromArrayList_0() {
  Task myTask = new Task("Mow the lawn");
  Task.clear();
  assertEquals(Task.all().size(), 0);
}
...

    After confirming this test fails, we may add the necessary logic:

    to-do/src/main/java/Task.java
    ...
public static void clear() {
  instances.clear();
}
...

    Remember, clear() is static because clearing all Task objects will occur on the class-level, not the instance-level. An individual Task object doesn't know about other Tasks; only about itself. Only the class can access or interact with all objects at once.

    Assigning Unique IDs

    Now that we can manage a list of many Tasks, let's add a unique ID to each. This will not only allow us to differentiate between similar-looking tasks, but it will later assist in locating a specific Task, and routing in Spark.

    Eventually, we'll want users to be able to click an individual Task to view its details. That means each Task will require its own page in our app. Later on, we'll also use the ID property to create unique URLs for each task's page.

    First, let's implement logic to assign and access IDs; beginning with a test:

    to-do/src/test/java/TaskTest.java
    ...
@Test
public void getId_tasksInstantiateWithAnID_1() {
  Task.clear();  // Remember, the test will fail without this line! We need to empty leftover Tasks from previous tests!
  Task myTask = new Task("Mow the lawn");
  assertEquals(1, myTask.getId());
}
...

    The Importance of Clearing After Tests

    Again, notice how we included Task.clear() at the beginning of the test. Again, this is very important. The list containing all Tasks will not automatically empty itself between each test. If we forget this line, we'll run into an error like this:

    java.lang.AssertionError: expected:<1> but was:<5>

    Each test creates one or more Task objects. And each time a Task is created, it's automatically saved to our list. So, even though this particular test only creates one Task, the list will contain "leftover" Tasks created by previous tests. The error message above informs us the test anticipatedinstances to contain 1 item, but it actually contained five.

    Thankfully, it's easy to avoid these issues by clearing the list at the start of our test. In an upcoming lesson, we'll learn how to clear it automatically, too.

    Moving on, we'll add the logic to assign and retrieve unique ID's:

    to-do/src/main/java/Task.java
    import java.util.ArrayList;
import java.time.LocalDateTime;

public class Task {
  private String mDescription;
  private boolean mCompleted;
  private LocalDateTime mCreatedAt;
  private static ArrayList<Task> instances = new ArrayList<Task>();
  private int mId;

  public Task(String description) {
    mDescription = description;
    mCompleted = false;
    mCreatedAt = LocalDateTime.now();
    instances.add(this);
    mId = instances.size();
  }

  public String getDescription() {
    return mDescription;
  }

  public boolean isCompleted() {
    return mCompleted;
  }

  public LocalDateTime getCreatedAt() {
    return mCreatedAt;
  }

  public static ArrayList<Task> all() {
    return instances;
  }

  public static void clear() {
    instances.clear();
  }

  public int getId() {
    return mId;
  }
}

    We want our application to assign the Task ID's automatically, instead of requiring the user to manually insert an ID number. That wouldn't be very user-friendly. So, similar to the fashion we automatically set each Task's mCompleted property to false, we add code to assign a unique mId to each Task in the constructor.

    Our getId() method test should be passing since we're clearing our list before running the test. The first Task created should be assigned an mId of 1. We're assigning the ID in the constructor with this line:

    mId = instances.size();

    After the first Task is added, instances.size() should be 1, after the second Task is created and added to the list it should be 2, and so on. By using instances.size() to assign mId, we ensure that each Task always has a unique id. Pretty cool, right?

    Finding Specific Objects

    So, we're currently adding Tasks to a list and automatically assigning them ID's. Nice work! But we'll also need a manner of locating specific Tasks using this unique ID. First, another test:

    to-do/src/test/java/TaskTest.java
    ...
@Test
public void find_returnsTaskWithSameId_secondTask() {
  Task firstTask = new Task("Mow the lawn");
  Task secondTask = new Task("Buy groceries");
  assertEquals(Task.find(secondTask.getId()), secondTask);
}
...

    And, the code we'll implement to make it pass:

    to-do/src/main/java/Task.java
    ...
public static Task find(int id) {
  return instances.get(id - 1);
}
...

    Here we've created another static method to locate a Task using its mId property. This method is static because it must sift through all Task objects to find the specific one we're seeking. And, because it is static, we will call it on the entire class, like this: Task.find().

    Again, an individual Task does not "know" about all other Tasks. It only knows its own properties. So, if a method's functionality must access all instances of a class, like our find() method above, it needs to be declared static.

    Terminology

     

     

    • Static: A variable type that is associated with the class itself, and not a specific instance of the class. For instance, if we wanted to maintain a list of all instances of a class, we could declare a static ArrayList in that class, and add each new instance we create into the ArrayList.

    Tips

     

     

    • An individual object does not "know" about all other instances of its class. It knows only its ownproperties. So, if a method's functionality must access all instances of a class it needs to be declared static.

    • The LocalDateTime class can automatically generate timestamps in a variety of formats for us. For instance, LocalDateTime.now() returns a timestamp of the current time.

    • Chaining .getDayOfWeek() after LocalDateTime.now() returns the specific day of the week denoted by the timestamp.
